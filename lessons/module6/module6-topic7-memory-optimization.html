Topic 7

Memory Optimization

Understanding and optimizing memory usage in Flutter apps — from garbage collection to detecting memory leaks.

Memory View Docs Memory Guide
Why Memory Matters

Memory optimization is crucial for building stable and performant Flutter apps. Poor memory management leads to:

💥
Out of Memory Crashes
App crashes when memory limit is exceeded
🐌
Slow Performance
Excessive garbage collection slows the app
📱
Device Slowdown
App makes the entire device sluggish
🔋
Battery Drain
Excessive memory usage drains battery

⚠️ Signs of Memory Issues

  • App crashes with OutOfMemoryError
  • App becomes slow over time (performance degradation)
  • Device becomes unresponsive
  • Memory usage grows progressively (leak)
  • Frequent garbage collection events
Dart Memory Management

Dart objects live in a portion of memory called the heap . The Dart VM allocates memory for objects at creation and releases it when objects are no longer used through garbage collection (GC) .

🧠 Key Concepts

  • Heap – Memory area where Dart objects live
  • Root object – Starting point for garbage collection
  • Reachability – Objects reachable from root won't be collected
  • Retaining path – Chain of references from root to an object
  • Garbage Collection (GC) – Automatic memory cleanup
Understanding Reachability 
class Child {}

class Parent {
  Child? child;
}

Parent parent1 = Parent();

void myFunction() {
  Child? child = Child();

  // child is reachable via one retaining path

  // root → myFunction → child

  Parent? parent2 = Parent()..child = child;
  parent1.child = child;

  // child now has THREE retaining paths:

  // root → myFunction → child

  // root → myFunction → parent2 → child

  // root → parent1 → child

  child = null;
  parent1.child = null;
  parent2 = null;

  // child is now UNREACHABLE → will be garbage collected
}

Shallow vs Retained Size

  • Shallow size – Only the object itself and its references
  • Retained size – Object + all objects it references
Understanding Sizes 
class Child {
  final hugeInstance = MyHugeInstance();
}

class Parent {
  Child? child;
}

Parent parent = Parent()..child = Child();

// Shallow size of Parent: only the reference to Child

// Retained size of Parent: Parent + Child + hugeInstance
Memory Leaks

A memory leak occurs when objects are no longer needed but remain reachable , preventing garbage collection. This causes memory usage to grow progressively until the app crashes.

⚠️ Common Leak Patterns

  • Closures – Capturing large objects in callbacks
  • BuildContext – Storing context in long-living objects
  • Stream subscriptions – Not canceling subscriptions
  • Animation controllers – Not disposing controllers
  • Global/static variables – Storing references that outlive objects

Closure Leaks

❌ BAD: Closure captures large object 

  // myHugeObject is captured by the closure
  final handler = () => print(myHugeObject.name);
  setHandler(handler); // myHugeObject stays alive!
✅ GOOD: Extract value before closure 

  // Extract only what's needed
  final name = myHugeObject.name;
  final handler = () => print(name);
  setHandler(handler);

BuildContext Leaks

❌ BAD: Context captured in closure 

  @override
  Widget build(BuildContext context) {
    // Closure captures the large, short-living context
    final handler = () => apply(Theme.of(context));
    useHandler(handler); // Might store it in long-living area
  }
✅ GOOD: Use long-living theme object 

  @override
  Widget build(BuildContext context) {
    // Theme is long-living, shared between contexts
    final theme = Theme.of(context);
    final handler = () => apply(theme);
    useHandler(handler);
  }

💡 BuildContext Rule

If a closure doesn't outlive the widget, it's safe to capture context . But never store context in a StatefulWidget's state fields — State outlives the widget and can cause leaks.

Memory Leak vs Memory Bloat

Memory Leak

  • Progressive memory increase
  • Objects never freed
  • Even small leaks → crash eventually
  • Example: Listener not disposed
  • Harder to detect

Memory Bloat

  • Using more memory than necessary
  • Objects are freed, but usage is high
  • Large images, oversized caches
  • Example: Overly large images
  • Easier to detect

⚠️ Which Is More Dangerous?

Leaks are more dangerous because even a small leak, repeated many times, will eventually cause an OutOfMemoryError . Always prioritize finding and fixing leaks.

Using the Memory View

DevTools provides a Memory view with several tabs to help you investigate memory issues.

📊
Expandable Chart
High-level memory allocation trace
📋
Profile Memory
Current memory allocation by class
🔄
Diff Snapshots
Compare memory before/after actions
🔍
Trace Instances
Trace allocations for specific classes

Chart Overview

The memory chart displays several metrics:

  • Dart/Flutter Heap – Objects in the heap
  • Dart/Flutter Native – Memory outside the heap (images, files)
  • Raster Cache – Flutter engine's rendering cache
  • RSS – Resident Set Size (total memory used)
  • GC events – Garbage collection activity

Profile Memory Tab

Shows current memory allocation by class . Use this to:

  • See which classes are using the most memory
  • Identify unexpected object counts
  • Track memory usage over time

Diff Snapshots Tab

The most powerful tool for finding leaks:

  • Take a snapshot before an action
  • Perform the action
  • Take a snapshot after
  • Diff shows what leaked
Preventing Memory Leaks

1. Always Dispose Controllers

✅ Correct Disposal Pattern 
class MyWidget extends StatefulWidget {
  @override
  _MyWidgetState createState() => _MyWidgetState();
}

class _MyWidgetState extends State<MyWidget>
    with SingleTickerProviderStateMixin {
  late AnimationController _controller;

  late StreamSubscription _subscription;

  @override
  void initState() {
    super.initState();
    _controller = AnimationController(vsync: this);
    _subscription = someStream.listen((data) {});
  }

  @override
  void dispose() {
    // ALWAYS dispose in reverse order

    _subscription.cancel();
    _controller.dispose();

    super.dispose();
  }
}

2. Cancel Stream Subscriptions

✅ Stream Subscription Management 
class MyService {
  StreamSubscription? _subscription;

  void startListening() {
    _subscription = someStream.listen((data) {
      // Handle data
    });
  }

  void dispose() {
    _subscription?.cancel();
    _subscription = null;
  }
}

3. Use Weak References

Using WeakReference for Callbacks 
class MyWidget extends StatefulWidget {
  @override
  _MyWidgetState createState() => _MyWidgetState();
}

class _MyWidgetState extends State<MyWidget> {
  void setupCallback() {
    // Use a WeakReference to avoid leaks

    final weakThis = WeakReference(this);
    someService.setCallback(() {
      final state = weakThis.target;

      if (state != null) {
        // State is still alive

        state.doSomething();
      }
    });
  }
}

4. Avoid Global Variables

❌ Bad: Global variable holds reference 
// ❌ Global variable - object never dies

final myGlobalList = <Widget>[];

// ✅ Good: Local or scoped variable

void myFunction() {
  final localList = <Widget>[];

  // Freed when function exits
}
Step-by-Step Memory Audit

Follow these steps to audit and optimize memory usage:

1.
Run in profile mode – Use flutter run --profile
2.
Open Memory view – Launch DevTools and go to Memory tab
3.
Take baseline snapshot – Capture current memory state
4.
Perform action – Navigate, scroll, or interact with your app
5.
Take second snapshot – Capture memory after the action
6.
Diff snapshots – Look for objects that should have been freed
7.
Trace leaks – Use retaining paths to find why objects are held
8.
Fix and re-test – Apply fixes and verify memory is freed
Common Mistakes
❌ Mistake 1: Not disposing controllers

AnimationController, TextEditingController, and other controllers hold resources that must be released.

✅ Correct: Always dispose in dispose()

Call dispose() on all controllers in the State's dispose() method.

❌ Mistake 2: Storing BuildContext in fields

Storing context in a State field prevents garbage collection.

✅ Correct: Use context only in build

Pass context to methods that need it, don't store it as a field.

❌ Mistake 3: Not canceling stream subscriptions

Active stream subscriptions keep the listener alive, preventing garbage collection.

✅ Correct: Cancel subscriptions in dispose

Always call cancel() on stream subscriptions in dispose() .

❌ Mistake 4: Using global lists/collections

Global variables keep objects alive forever, causing memory leaks.

✅ Correct: Use local variables

Prefer local variables or well-scoped collections that can be cleared when no longer needed.

🎯 Key Takeaway

Memory optimization is essential for building stable Flutter apps. Understand Dart's garbage collection — objects are freed when they become unreachable . Prevent memory leaks by disposing controllers, canceling subscriptions, and avoiding storing BuildContext or large objects in closures. Use DevTools' Memory view with Diff Snapshots to detect and fix leaks. Remember: even a small leak, repeated many times, will eventually crash your app with an OutOfMemoryError .
