Adaptive Design

The 3-step approach to making your Flutter app beautiful on any device.

Adaptive Design Docs Adaptive & Responsive
What Is Adaptive Design?

Adaptive design is about making your app usable in the available space. While responsive design focuses on fitting the UI into the space, adaptive design focuses on selecting the right UI for the space.

💡 The Key Question

Should a tablet UI use bottom navigation or side-panel navigation? This is the kind of question adaptive design answers — it's not about making the same layout smaller, but about choosing the right layout for the device.

Usability First

Device Intelligence
Know what device the user is on

📱 Responsive

  • Same layout, different sizes
  • Elements resize and reposition
  • "Fitting" the UI
  • Example: Text scales, columns wrap

🔄 Adaptive

  • Different layouts for different sizes
  • UI changes based on context
  • "Usability" in the space
  • Example: Bottom nav → Side nav
The 3-Step Approach

Google engineers recommend a 3-step approach for making apps adaptive:

Step 1
📝
Abstract
Identify widgets that need to be dynamic and abstract out shared data
Step 2
📐
Measure
Determine the size of your display area using MediaQuery or LayoutBuilder
Step 3
🔀
Branch
Choose the right UI version based on the measured size
Step 1: Abstract

First, identify the widgets that you plan to make dynamic. Analyze the constructors for those widgets and abstract out the data that you can share.

🔍 Common Adaptive Widgets

  • Dialogs – Fullscreen vs modal
  • Navigation – Bottom bar vs rail
  • Layout – Tall vs wide arrangements
  • Lists – Single column vs multi-column

Step 2: Measure

You have two ways to determine the size of your display area: MediaQuery and LayoutBuilder .

MediaQuery

  • Size of the entire app window
  • Best for fullscreen widgets
  • Rebuilds when window size changes
  • Use MediaQuery.sizeOf(context)

LayoutBuilder

  • Size constraints from parent
  • Best for nested widgets
  • Only rebuilds when parent constraints change
  • Use constraints.maxWidth
Measuring with MediaQuery class AdaptiveWidget extends StatelessWidget { @override Widget build (BuildContext context) { // Get the full screen size final width = MediaQuery. sizeOf (context).width; // Choose layout based on screen width if (width < 600 ) { return MobileLayout(); } else if (width < 1200 ) { return TabletLayout(); } else { return DesktopLayout(); } } }
Measuring with LayoutBuilder class AdaptiveContainer extends StatelessWidget { @override Widget build (BuildContext context) { return LayoutBuilder( builder: (context, constraints) { // Get the available space from parent final maxWidth = constraints.maxWidth; // Choose layout based on available space if (maxWidth < 400 ) { return _buildCompactLayout(); } else if (maxWidth < 800 ) { return _buildMediumLayout(); } else { return _buildExpandedLayout(); } }, ); } }

⚠️ Important: Don't Lock Orientation

When making adaptive apps, don't lock your app to portrait mode . If your app runs in a small window on a large screen and you've portrait-locked it, the app window locks to the center of the screen with black bars — a poor UX. Instead, adapt to the available space.

Step 3: Branch

Now decide what breakpoints to use when choosing which UI version to display. Your choice shouldn't depend on the type of device, but on the available window size .

Branching Logic for Navigation class AdaptiveNavigation extends StatefulWidget { @override _AdaptiveNavigationState createState () => _AdaptiveNavigationState(); } class _AdaptiveNavigationState extends State<AdaptiveNavigation> { int _selectedIndex = 0 ; final List<Widget> _pages = [HomePage(), SearchPage(), ProfilePage()]; final List<NavigationDestination> _destinations = [ NavigationDestination(icon: Icon(Icons.home), label: 'Home' ), NavigationDestination(icon: Icon(Icons.search), label: 'Search' ), NavigationDestination(icon: Icon(Icons.person), label: 'Profile' ), ]; @override Widget build (BuildContext context) { final width = MediaQuery. sizeOf (context).width; // Branch: Choose navigation based on width return Scaffold( body: Row( children: [ // Use NavigationRail for larger screens if (width >= 600 ) NavigationRail( selectedIndex: _selectedIndex, onDestinationSelected: (index) { setState (() => _selectedIndex = index); }, labelType: NavigationRailLabelType.all, destinations: _destinations, ), Expanded(child: _pages[_selectedIndex]), ], ), // Use NavigationBar for smaller screens bottomNavigationBar: width < 600 ? NavigationBar( selectedIndex: _selectedIndex, onDestinationSelected: (index) { setState (() => _selectedIndex = index); }, destinations: _destinations, ) : null , ); } }
Branching Logic for Dialogs void showAdaptiveDialog (BuildContext context) { final width = MediaQuery. sizeOf (context).width; // Branch: Choose dialog style based on width if (width < 600 ) { // Mobile: Fullscreen bottom sheet showModalBottomSheet( context: context, isScrollControlled: true , builder: (context) => DialogContent( title: 'Action' , message: 'Confirm this action?' , onConfirm: () => Navigator. pop (context), ), ); } else { // Tablet/Desktop: Modal dialog showDialog( context: context, builder: (context) => AlertDialog( title: Text( 'Action' ), content: DialogContent( title: 'Action' , message: 'Confirm this action?' , onConfirm: () => Navigator. pop (context), ), ), ); } }

📊 Recommended Breakpoints

  • < 600 dp – Mobile (compact): Bottom navigation, single column
  • 600 - 840 dp – Tablet (medium): Navigation rail, two columns
  • > 840 dp – Desktop (expanded): Side navigation, multiple columns

Step-by-Step Implementation

Follow these steps to make your app adaptive:

1.
Identify adaptive widgets – Find widgets that need different versions (navigation, dialogs, layout)
2.
Abstract shared data – Extract data that can be used across all versions
3.
Choose measurement method – Use MediaQuery for fullscreen, LayoutBuilder for nested
4.
Define breakpoints – Set widths for mobile, tablet, and desktop
5.
Branch your UI – Use if statements to choose the right layout
6.
Test all sizes – Use device preview and resize your browser
7.
Refine and polish – Adjust interactions, spacing, and animations for each size
Common Mistakes
❌ Mistake 1: Locking the app to portrait mode

Portrait locking prevents the app from adapting to different window sizes on large screens.

✅ Correct: Allow all orientations

Let the app adapt to any window size. Use MediaQuery to detect and respond to changes.

❌ Mistake 2: Duplicating data across layouts

Defining the same data (navigation items, content) separately for each layout leads to bugs.

✅ Correct: Abstract and share data

Define data once and reuse it across all layout versions. This ensures consistency.

❌ Mistake 3: Only testing on one device

Testing only on your phone means you miss tablet and desktop issues.

✅ Correct: Test multiple sizes

Use Flutter's device preview, emulators, or resize your browser window to test all sizes.

❌ Mistake 4: Forgetting about foldables

Foldable devices have unique layouts (dual screen, hinge). Ignoring them misses a growing market.

✅ Correct: Support foldables

Use DisplayFeature and foldable APIs for dual-screen and foldable devices.

🎯 Key Takeaway

Adaptive design is about making your app usable in any space, not just fitting it. Use the 3-step approach : Abstract shared data, Measure the available space, and Branch to the right UI. Always test on multiple screen sizes and don't lock orientation — let your app adapt to the user's available window. With these principles, your app will be beautiful and usable on any device.
