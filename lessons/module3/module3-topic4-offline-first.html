Offline-First Architecture

Building resilient Flutter apps that work seamlessly with or without an internet connection.

Flutter Docs – App Architecture
What Is Offline-First?

Offline-first is an architectural approach where your app is designed to work without an internet connection as the primary use case, rather than treating offline as an edge case. This means:

  • Data is stored locally first (on the device)
  • The app reads from and writes to local storage
  • Background synchronization happens when the network is available
  • The user experience remains consistent regardless of connectivity

✅ Benefits of Offline-First

  • Reliability – Works anywhere, anytime
  • Speed – Local data access is instant
  • Data usage – Reduced network calls
  • User experience – No loading spinners or "no connection" errors
  • Scalability – Reduced server load
Core Principles

Offline-first architecture is built on four core principles:

💾
Local First
Always read from and write to local storage first
🔄
Sync in Background
Synchronize with the server when online
⚡
Optimistic Updates
Update UI immediately, sync later
Synchronization Strategies

There are three main strategies for synchronizing data between local storage and the server:

🔄 Pull-based Sync

How it works: The app periodically checks the server for new data and downloads it.

Best for: Reference data, read-only content

Example: News app that refreshes every hour

📤 Push-based Sync

How it works: The app sends local changes to the server immediately when online.

Best for: User-generated content, real-time updates

Example: Notes app that syncs changes as you type

🔁 Bi-directional Sync

How it works: Both pull and push synchronization working together.

Best for: Collaborative apps, two-way data flow

Example: Task management app with team members

📱 Conflict Resolution

How it works: Rules for handling conflicting changes from multiple sources.

Strategies: Last write wins, merge changes, manual resolution

Example: Git-like merge for collaborative editing

Implementation Patterns

Here are the key patterns for implementing offline-first architecture in Flutter.

lib/services/offline_service.dart – Offline Service 
import 'dart:async';

import 'package:connectivity_plus/connectivity_plus.dart';

import 'package:hive_flutter/hive_flutter.dart';

import 'package:http/http.dart' as http;

class SyncQueueItem {
  final String id;

  final String collection;

  final String operation;

  // 'create', 'update', 'delete'

  final Map<String, dynamic> data;

  final DateTime timestamp;

  SyncQueueItem({
    required this.id,

    required this.collection,

    required this.operation,

    required this.data,

    DateTime? timestamp,
  }) : this.timestamp = timestamp ?? DateTime.now();
}

class OfflineService {
  final http.Client client;

  final Box _syncQueue;

  final Connectivity _connectivity = Connectivity();

  OfflineService({http.Client? client, required Box syncQueue})
    : client = client ?? http.Client(),
      _syncQueue = syncQueue;

  // ----- CHECK CONNECTIVITY -----

  Future<bool> isOnline() async {
    final result = await _connectivity.checkConnectivity();

    return result != ConnectivityResult.none;
  }

  // ----- QUEUE OPERATIONS FOR SYNC -----

  void queueOperation({
    required String collection,

    required String operation,

    required Map<String, dynamic> data,
  }) {
    final id = DateTime.now().millisecondsSinceEpoch.toString();

    final item = SyncQueueItem(
      id: id,
      collection: collection,
      operation: operation,
      data: data,
    );
    _syncQueue.put(id, {
      'collection': collection,

      'operation': operation,

      'data': data,

      'timestamp': item.timestamp.toIso8601String(),
    });
    _attemptSync();
  }

  // ----- SYNC WITH SERVER -----

  Future<void> _attemptSync() async {
    if (!await isOnline()) return;

    final queueItems = _syncQueue.values.toList();

    for (var item in queueItems) {
      try {
        final collection = item['collection'] as String;

        final operation = item['operation'] as String;

        final data = item['data'] as Map<String, dynamic>;

        final key = _syncQueue.keyAt(_syncQueue.values.toList().indexOf(item));

        switch (operation) {
          case 'create':
            await _syncCreate(collection, data);

            break;

          case 'update':
            await _syncUpdate(collection, data);

            break;

          case 'delete':
            await _syncDelete(collection, data);

            break;
        }

        // Remove from queue after successful sync

        await _syncQueue.delete(key);
      } catch (e) {
        // If sync fails, keep in queue for later retry

        print('Sync failed: $e');
      }
    }
  }

  Future<void> _syncCreate(String collection, Map<String, dynamic> data) async {
    // Send POST request to server

    final response = await client.post(
      Uri.parse('https://api.example.com/api/$collection'),
      headers: {'Content-Type': 'application/json'},
      body: jsonEncode(data),
    );

    if (response.statusCode != 201) {
      throw Exception('Failed to create');
    }
  }

  Future<void> _syncUpdate(String collection, Map<String, dynamic> data) async {
    final id = data['id'];

    final response = await client.put(
      Uri.parse('https://api.example.com/api/$collection/$id'),
      headers: {'Content-Type': 'application/json'},
      body: jsonEncode(data),
    );

    if (response.statusCode != 200) {
      throw Exception('Failed to update');
    }
  }

  Future<void> _syncDelete(String collection, Map<String, dynamic> data) async {
    final id = data['id'];

    final response = await client.delete(
      Uri.parse('https://api.example.com/api/$collection/$id'),
      headers: {'Content-Type': 'application/json'},
    );

    if (response.statusCode != 200 && response.statusCode != 204) {
      throw Exception('Failed to delete');
    }
  }

  // ----- FORCE SYNC -----

  Future<void> forceSync() async {
    await _attemptSync();
  }
}

Step-by-Step Explanation
1.
Local Storage First – Always write to local storage (Hive) first. This ensures the user sees their data immediately.
2.
Check Connectivity – Use connectivity_plus to check if the device is online.
3.
Sync if Online – If connected, send changes to the server immediately and mark as synced.
4.
Queue if Offline – If offline, store operations in a sync queue for later processing.
5.
Background Sync – When connectivity is restored, process all queued operations.
Best Practices

✅ Do's

  • Always read from local storage first
  • Use optimistic updates for better UX
  • Store timestamps for conflict resolution
  • Use a sync queue for offline operations
  • Show sync status to users (synced/pending)
  • Handle conflicts with clear rules
  • Use background sync when connectivity is restored

❌ Don'ts

  • Don't block the UI during sync operations
  • Don't ignore conflicts – they will happen
  • Don't sync large files in the background without user consent
  • Don't lose data if sync fails – keep it in the queue
  • Don't over-sync – use debouncing for frequent changes
Common Mistakes
❌ Mistake 1: Not handling conflicts

When multiple devices edit the same data, conflicts can occur. Always have a conflict resolution strategy.

✅ Correct: Use conflict resolution

Strategies: Last Write Wins, Merge, or Manual Resolution with user input.

❌ Mistake 2: Syncing too frequently

Syncing on every keystroke or network change can drain battery and use data.

✅ Correct: Use debouncing

Wait for the user to stop typing (500ms-1s) before syncing changes.

❌ Mistake 3: Not showing sync status

Users need to know if their data is synced or pending. Confusion leads to frustration.

✅ Correct: Show sync status

Use icons, colors, or text to indicate sync status (Synced, Pending, Error).

🎯 Key Takeaway

Offline-first architecture is about designing for resilience . Always read from and write to local storage first , then sync with the server when connected. Use a sync queue for offline operations and conflict resolution for consistency. Your users will thank you for a reliable, fast experience anywhere.
