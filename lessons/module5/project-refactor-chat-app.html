Refactor Chat App

Apply Clean Architecture principles to refactor the real-time Chat App into a maintainable, testable, and scalable Flutter application.

Refactoring Project Realtime Flutter Architecture Guide GetIt Package Supabase Realtime
Project Overview

In this project, you'll refactor the Chat App built in Module 4 (Realtime Chat App Project) using Clean Architecture principles. The Chat App uses Supabase Realtime for live messaging, making it an excellent candidate for applying Clean Architecture patterns to handle real-time data streams.

📚 Prerequisites

  • Completed Realtime Chat App Project from Module 4
  • Understanding of Architecture Overview
  • Understanding of Repository Pattern
  • Understanding of Dependency Injection
  • Understanding of Feature-First Architecture
  • Familiarity with Supabase Realtime subscriptions

🎯 Learning Objectives

  • Apply Clean Architecture to a real-time application
  • Implement Repository Pattern with real-time data streams
  • Handle Supabase Realtime subscriptions in the repository layer
  • Use Stream-based state management in the ViewModel
  • Set up dependency injection with GetIt for real-time services
  • Organize code using Feature-First Architecture
Before vs After

Let's compare the original Chat App structure with the refactored version:

❌ Before (Module 4)

  • Supabase subscriptions in widget code
  • UI and real-time logic tightly coupled
  • No clear separation of concerns
  • Hard to test (can't mock Supabase)
  • No abstraction for real-time data
  • Code scattered across files
  • Difficult to switch backends

✅ After (Clean Architecture)

  • Supabase subscriptions in data sources
  • UI separated from real-time logic
  • Clear data/domain/presentation separation
  • Easy to test with mock repositories
  • Stream-based data flow through repository
  • Feature-based organization
  • Backend-agnostic domain layer

Move Supabase interactions to data sources:

lib/features/auth/data/datasources/auth_remote_datasource.dart import 'package:supabase_flutter/supabase_flutter.dart'; class AuthRemoteDataSource { final SupabaseClient client; AuthRemoteDataSource(this.client); Future<Map<String, dynamic>> login(String email, String password) async { final response = await client.auth.signInWithPassword( email: email, password: password, ); return response.user!.toJson(); } Future<Map<String, dynamic>> register(String email, String password) async { final response = await client.auth.signUp( email: email, password: password, ); return response.user!.toJson(); } Future<void> logout() async { await client.auth.signOut(); } Map<String, dynamic>? getCurrentUser() { final user = client.auth.currentUser; return user?.toJson(); } }

Step 4: Implement Repositories

Create repository interfaces and implementations:

lib/features/chat/domain/repositories/chat_repository.dart import '../models/message.dart'; import 'dart:async'; abstract class ChatRepository { Future<List<Message>> getMessages(); Future<Message> sendMessage(Message message); Stream<List<Message>> subscribeToMessages(); void dispose(); }
lib/features/chat/data/repositories/chat_repository_impl.dart import 'dart:async'; import '../../domain/repositories/chat_repository.dart'; import '../../domain/models/message.dart'; import '../datasources/chat_remote_datasource.dart'; class ChatRepositoryImpl implements ChatRepository { final ChatRemoteDataSource remoteDataSource; final StreamController<List<Message>> _messageStreamController = StreamController<List<Message>>.broadcast(); List<Message> _messages = []; ChatRepositoryImpl(this.remoteDataSource) { // Start listening to real-time updates _listenToRealtime(); } void _listenToRealtime() { remoteDataSource.subscribeToMessages().listen((newMessages) { // Update the message list _messages = newMessages .map((data) => Message.fromJson(data)) .toList(); _messageStreamController.add(_messages); }); } @override Future<List<Message>> getMessages() async { if (_messages.isNotEmpty) { return _messages; } final data = await remoteDataSource.getMessages(); _messages = data.map((json) => Message.fromJson(json)).toList(); return _messages; } @override Future<Message> sendMessage(Message message) async { final data = await remoteDataSource.sendMessage(message.toJson()); final newMessage = Message.fromJson(data); // Add to local list and notify listeners _messages = [..._messages, newMessage]; _messageStreamController.add(_messages); return newMessage; } @override Stream<List<Message>> subscribeToMessages() { return _messageStreamController.stream; } @override void dispose() { _messageStreamController.close(); } }

Step 6: Create ViewModels

Manage state and interact with use cases using Stream-based state management:

lib/features/chat/presentation/viewmodels/chat_viewmodel.dart import 'dart:async'; import 'package:flutter_riverpod/flutter_riverpod.dart'; import '../../../domain/usecases/send_message_usecase.dart'; import '../../../domain/usecases/subscribe_to_messages_usecase.dart'; import '../../../domain/models/message.dart'; class ChatViewModel extends StateNotifier<ChatState> { final SendMessageUseCase sendMessageUseCase; final SubscribeToMessagesUseCase subscribeToMessagesUseCase; StreamSubscription<List<Message>>? _subscription; ChatViewModel({ required this.sendMessageUseCase, required this.subscribeToMessagesUseCase, }) : super(ChatState.initial()) { _listenToMessages(); } void _listenToMessages() { _subscription = subscribeToMessagesUseCase.execute().listen( (messages) { state = state.copyWith( messages: messages, isLoading: false, error: null, ); }, onError: (error) { state = state.copyWith( isLoading: false, error: error.toString(), ); }, ); } Future<void> sendMessage({ required String content, required String userId, required String userName, }) async { try { await sendMessageUseCase.execute( content: content, userId: userId, userName: userName, ); } catch (e) { state = state.copyWith(error: e.toString()); } } @override void dispose() { _subscription?.cancel(); super.dispose(); } } class ChatState { final bool isLoading; final List<Message> messages; final String? error; ChatState({ this.isLoading = false, this.messages = const [], this.error, }); factory ChatState.initial() => ChatState(isLoading: true); ChatState copyWith({ bool? isLoading, List<Message>? messages, String? error, }) { return ChatState( isLoading: isLoading ?? this.isLoading, messages: messages ?? this.messages, error: error ?? this.error, ); } }

Handling Real-Time Streams

One of the key challenges in the Chat App is handling real-time data streams. Here's how Clean Architecture helps manage this complexity:

💡 Stream Management Strategy

  • Data Source – Creates the Supabase Realtime subscription
  • Repository – Maintains the stream and message state
  • Use Case – Provides the stream to the ViewModel
  • ViewModel – Listens to the stream and updates UI state
  • UI – Reacts to state changes from the ViewModel
lib/features/chat/data/datasources/chat_remote_datasource.dart – Stream Implementation // The data source returns a Stream of raw data Stream<List<Map<String, dynamic>>> subscribeToMessages() { final controller = StreamController<List<Map<String, dynamic>>>.broadcast(); final subscription = client .channel('realtime:messages') .onPostgresChanges( event: PostgresChangeEvent.all, schema: 'public', table: 'messages', callback: (payload) { // When a change happens, fetch all messages _fetchAllMessages().then((messages) { controller.add(messages); }); }, ) .subscribe(); // Clean up on cancel controller.onCancel = () { subscription.unsubscribe(); }; return controller.stream; }

✅ Stream Management Benefits

  • Clean separation – Stream logic isolated in data layer
  • Testability – Streams can be mocked in tests
  • Memory safety – Proper disposal of subscriptions
  • Reactive UI – UI automatically updates with new messages
  • Backend agnostic – Domain layer doesn't know about Supabase

Testing the Refactored App

Testing real-time applications is easier with Clean Architecture because you can mock the stream at the repository level:

test/features/chat/domain/usecases/send_message_usecase_test.dart import 'package:test/test.dart'; import 'package:mockito/mockito.dart'; import '../../../../lib/features/chat/domain/usecases/send_message_usecase.dart'; import '../../../../lib/features/chat/domain/repositories/chat_repository.dart'; import '../../../../lib/features/chat/domain/models/message.dart'; class MockChatRepository extends Mock implements ChatRepository {} void main() { late MockChatRepository mockRepository; late SendMessageUseCase useCase; setUp(() { mockRepository = MockChatRepository(); useCase = SendMessageUseCase(mockRepository); }); test('sends a message successfully', () async { // Arrange final expectedMessage = Message( id: '1', content: 'Hello World', userId: 'user1', userName: 'Alice', timestamp: DateTime.now(), ); when(mockRepository.sendMessage(any)) .thenAnswer((_) async => expectedMessage); // Act final result = await useCase.execute( content: 'Hello World', userId: 'user1', userName: 'Alice', ); // Assert expect(result, expectedMessage); verify(mockRepository.sendMessage(any)).called(1); }); test('throws ArgumentError when content is empty', () async { expect( () => useCase.execute( content: '', userId: 'user1', userName: 'Alice', ), throwsA(isA<ArgumentError>()), ); }); }

✅ Testing Benefits Achieved

  • Mock streams – Repository streams can be mocked for testing
  • Isolated tests – Use cases tested independently of real-time infrastructure
  • Edge cases – Easy to test error conditions in streams
  • Business logic – Tested without Supabase connection
